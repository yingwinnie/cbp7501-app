## 今日完成事項

### 1. Bug #1 修正：TC05 超長 Description 警告不觸發

**問題**：上傳包含超長描述的 PDF 後，編輯頁的「字數過多」警告沒有出現。

**根本原因**：原本 `handlePDFUpload` 只做格式驗證，沒有實際讀取 PDF 文字內容來填入 `lineItem.description`。

**修法**：
- 新增 `parsePDFText(file)` 函式，從 CDN 動態載入 `pdfjs-dist v3.11.174`
- CDN 來源：`https://cdnjs.cloudflare.com/ajax/libs/pdf.js/3.11.174/pdf.min.js`
- 解析 PDF 文字後，篩選符合 CBP 7501 格式的行（正則：`S122|9903|OTHER ELEC|LAMPS?|LIGHT|TRANSIT|FITTNG|FITTING|9405`）
- 將抽取結果寫入 `lineItem.description`，超過 180 字即觸發警告

```typescript
const parsePDFText = async (file: File): Promise<string> => {
  const PDFJS_VERSION = "3.11.174";
  const CDN_BASE = `https://cdnjs.cloudflare.com/ajax/libs/pdf.js/${PDFJS_VERSION}`;
  if (!(window as any).pdfjsLib) {
    await new Promise<void>((resolve, reject) => {
      const s = document.createElement("script");
      s.src = `${CDN_BASE}/pdf.min.js`;
      s.onload = () => resolve();
      s.onerror = reject;
      document.head.appendChild(s);
    });
    (window as any).pdfjsLib.GlobalWorkerOptions.workerSrc =
      `${CDN_BASE}/pdf.worker.min.js`;
  }
  // ... 逐頁抽取文字
};
```

### 2. Bug #2 修正：TC10 損毀 PDF 未被阻擋

**問題**：上傳結構損毀的 PDF 後，系統仍然跳轉到編輯頁。

**修法（兩層驗證）**：

**第一層 — Magic Bytes 檢查**：
```typescript
const buf = await file.slice(0, 5).arrayBuffer();
const magic = new TextDecoder().decode(buf);
if (!magic.startsWith("%PDF-")) {
  showToast(`❌「${file.name}」格式錯誤，僅接受有效 PDF 檔案`);
  return false;
}
```

**第二層 — pdfjs 結構驗證**：
```typescript
} catch (err) {
  // pdfjs 解析失敗 → 結構損毀，阻擋上傳
  showToast(`❌「${file.name}」PDF 已損毀，無法讀取`);
  return false;
}
```

**發現原因**：`invalid_001/002/004/005.pdf` 雖然結構損毀，但前 5 bytes 仍是 `%PDF-`，所以第一層過得了。加上第二層（pdfjs 解析失敗即擋）後全部正確攔截。

## 重要技術決策

| 決策 | 選擇 | 原因 |
|------|------|------|
| pdfjs 載入方式 | CDN 動態載入 | `package.json` 沒有 pdfjs-dist，避免改動依賴 |
| 損毀 PDF 驗證 | 兩層（magic bytes + pdfjs） | 部分損毀 PDF 仍有合法 header，單靠 magic bytes 不夠 |
| TypeScript strict | 關閉（false） | 原 CodeSandbox 版本無 strict，避免 implicit any 編譯錯誤 |
| 部署平台 | Vercel | 免費、自動 CI/CD、與 GitHub 整合簡單 |
